## 命名规范

### 1. 文档类型标识

| 文档类型 | 前缀/后缀 | 示例 | 说明 |
|---------|----------|------|------|
| 计划文档 | `-plan` | `development-plan.md` | 开发计划、项目规划 |
| 指南文档 | `-guide` | `setup-guide.md` | 操作指南、使用手册 |
| 设计文档 | `-design` | `architecture-design.md` | 架构设计、技术设计 |
| 规范文档 | `-spec` | `coding-spec.md` | 规范说明、标准文档 |
| 报告文档 | `-report` | `progress-report.md` | 进度报告、分析报告 |
| 索引文档 | `README` | `README.md` | 目录索引、导航文档 |

### 2. 命名格式

```
{模块}-{类型}.{扩展名}
```

**示例**:
- `development-plan.md` (开发计划)
- `api-reference.md` (API参考)
- `database-design.md` (数据库设计)
- `coding-spec.md` (编码规范)

### 3. 特殊文档命名

| 用途 | 文件名 | 说明 |
|------|--------|------|
| 主索引 | `README.md` | 项目根目录和各子目录的导航索引 |
| 配置说明 | `CONFIGURATION.md` | 全大写字母，突出重要性 |
| 变更日志 | `CHANGELOG.md` | 记录版本变更历史 |
| 贡献指南 | `CONTRIBUTING.md` | 开源项目贡献说明 |

## 存储路径规范

### 1. 根目录文档

```
project-root/
├── README.md              # 项目主文档
├── CONFIGURATION.md       # 配置说明
├── CHANGELOG.md          # 变更日志
└── CONTRIBUTING.md       # 贡献指南
```

### 2. docs目录结构

```
docs/
├── README.md              # 文档中心首页
├── {category}/            # 分类目录
│   ├── README.md         # 分类索引
│   ├── {module}-plan.md  # 模块计划
│   ├── {module}-guide.md # 模块指南
│   └── {module}-spec.md  # 模块规范
├── archive/              # 归档目录
│   └── {version}/        # 按版本归档
└── templates/            # 文档模板
    ├── plan-template.md
    ├── guide-template.md
    └── spec-template.md
```

### 3. 分类目录规范

| 分类 | 目录名 | 包含文档类型 |
|------|--------|-------------|
| 架构设计 | `architecture/` | 架构图、设计文档、技术选型 |
| 开发规范 | `development/` | 开发计划、编码规范、流程指南 |
| 功能模块 | `modules/` | 各模块详细文档 |
| API文档 | `api/` | 接口说明、SDK文档 |
| 操作指南 | `guides/` | 使用手册、配置指南 |
| 参考资料 | `reference/` | 配置参数、FAQ、术语表 |
| 归档文档 | `archive/` | 历史版本、废弃文档 |

## 版本控制策略

### 1. 文档版本标记

```
{文档名}_v{主版本}.{次版本}.{修订版本}.{日期}
```

**示例**:
- `development-plan_v1.2.0_20260219.md`
- `api-reference_v2.0.1_20260220.md`

### 2. 版本更新规则

| 变更类型 | 版本号更新 | 说明 |
|---------|-----------|------|
| 重大变更 | 主版本+1 | 架构性调整、功能重构 |
| 功能更新 | 次版本+1 | 新增功能、重要改进 |
| 修复完善 | 修订版本+1 | 错误修正、细节优化 |
| 日常维护 | 日期更新 | 小幅调整、内容补充 |

### 3. Git分支策略

```
main                    # 主分支，稳定版本
├── docs/v1.0          # 文档v1.0版本分支
├── docs/v1.1          # 文档v1.1版本分支
└── docs/develop       # 文档开发分支
```

## 归档机制

### 1. 归档触发条件

- 文档版本重大升级（主版本变更）
- 功能模块废弃或重构
- 项目阶段性完成
- 超过6个月未更新的文档

### 2. 归档目录结构

```
docs/archive/
├── versions/                 # 按版本归档
│   ├── v1.0_202601/         # v1.0版本归档
│   └── v1.1_202606/         # v1.1版本归档
├── deprecated/              # 废弃文档
│   ├── old-modules/         # 废弃模块文档
│   └── legacy-systems/      # 遗留系统文档
└── historical/              # 历史文档
    ├── project-proposals/   # 项目提案
    └── design-evolution/    # 设计演进记录
```

### 3. 归档文件命名

```
{原文档名}_archived_{归档日期}_{原因}.{扩展名}
```

**示例**:
- `development-plan_archived_20260219_refactored.md`
- `old-api-reference_archived_20260115_superseded.md`

## 维护责任

### 1. 文档负责人制度

| 文档类别 | 负责人 | 职责 |
|---------|--------|------|
| 架构文档 | 架构师 | 设计文档维护、技术决策记录 |
| 开发规范 | 技术主管 | 编码规范更新、流程优化 |
| 功能模块 | 模块负责人 | 模块文档编写、API说明维护 |
| 操作指南 | 产品经理 | 用户手册更新、配置说明维护 |
| API文档 | 后端负责人 | 接口文档同步、SDK文档维护 |

### 2. 维护周期

| 文档类型 | 更新频率 | 审查周期 |
|---------|---------|---------|
| 计划文档 | 项目里程碑 | 每季度 |
| 规范文档 | 重大变更时 | 每半年 |
| 指南文档 | 功能更新时 | 每季度 |
| API文档 | 接口变更时 | 每月 |
| 索引文档 | 结构调整时 | 每月 |

## 实施指南

### 1. 迁移步骤

1. **现状评估**: 统计现有文档，识别命名不规范的文件
2. **重命名执行**: 按照新规范重命名所有文档
3. **链接更新**: 更新文档间的相互引用链接
4. **目录结构调整**: 按新规范组织目录结构
5. **版本控制**: 建立Git分支和标签管理机制

### 2. 质量检查清单

- [ ] 所有文档遵循命名规范
- [ ] 目录结构符合存储路径规范
- [ ] 重要文档有明确负责人
- [ ] 文档间引用链接有效
- [ ] 归档机制已建立并测试
- [ ] 版本控制策略已实施

### 3. 持续改进

- 每月进行文档质量评估
- 收集团队使用反馈
- 定期更新规范内容
- 建立文档贡献激励机制

### 常见问题

**Q: 如何处理跨模块的文档？**
A: 放在相关的主模块目录下，并在其他模块的README中添加引用链接。

**Q: 旧文档如何处理？**
A: 按归档机制移至archive目录，并在原位置添加指向归档文档的链接。

**Q: 文档更新流程是什么？**
A: 1) 在develop分支修改 → 2) 提交PR → 3) 审查通过后合并 → 4) 更新版本号 → 5) 推送至main分支
